# 版本控制

SeerAPI 及其 SDK 均采用**统一的语义化版本控制**策略。

为了解决游戏数据频繁变动带来的版本管理难题，并确保所有下游工具（SDK）能正确处理依赖关系，我们定义了基于**百分位主版本号**的 [SemVer](https://semver.org/) 规范。

## 版本号格式

版本号格式为 `Major.Minor.Patch`（例如 `103.2.0`）。

### 1. 主版本号 (Major)

主版本号采用三位数字格式 `Xxx`（例如 `103`、`104`、`200`），其变化代表**破坏性变更**。

主版本号的结构含义如下：

*   **百位数字 (X)**: **架构级破坏性变更**。
    *   代表 API 整体架构、URL 路径结构或核心模型的重写。
    *   **重要**：百位数字对应 API 的 URL 路径版本（例如 `v1` 对应 `1xx`，`v2` 对应 `2xx`）。
    *   *例如：从 `103` 升级到 `200`，意味着 API URL 从 `/v1/` 变为 `/v2/`。*
*   **后两位数字 (xx)**: **模型级破坏性变更**。
    *   代表具体数据模型（字段类型、名称）发生了不兼容的修改。
    *   *例如：从 `103` 升级到 `104`，意味着某个数据字段被移除或重命名了。*

### 2. 次版本号 (Minor)

代表**功能新增（向后兼容）**。

*   新增了数据模型。
*   在现有模型中新增了可选字段。
*   新增了 API 接口。

### 3. 修订号 (Patch)

仅预留给**工具链 / SDK**，用于表示代码更改，API定义的变更应至少是 Minor 级别。

*   修复或优化 SDK 代码逻辑。
*   修复文档生成工具的错误。
*   优化分析器性能。

## 示例演进

| 版本号 | 模块 | 变更类型 | 说明 | 兼容性 |
| :--- | :--- | :--- | :--- | :--- |
| **103.2.0** | API / Models | 初始状态 | 当前基准版本 | - |
| **103.2.1** | 其他模块 | Patch | 修复了一个分析器bug | 兼容 103.2.0 |
| **103.3.0** | API / Models | Minor | 新增了 `Pet.skin_id` 字段 | 兼容 103.2.0 |
| **104.0.0** | API / Models | **Major** | 移除了 `Skill.foo_bar` 字段 | **不兼容** 103.x |
| **104.0.1** | 其他模块 | Patch | 修复 104.0.0 的一个小 Bug | 兼容 104.0.0 |
| **200.0.0** | API / Models | **Major (架构)** | API 路径升级为 `/v2/`，重构所有模型 | **不兼容** 1xx |

## 适用范围

本规则适用于 SeerAPI monorepo 下的所有包。
